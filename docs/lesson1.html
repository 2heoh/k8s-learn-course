Сущности Kubernetes (что это такое)

Pod

Минимальная единица запуска. Обычно содержит 1 контейнер (но может быть несколько). У Pod есть IP внутри кластера, но он эфемерен — может пересоздаваться.

Deployment

Описывает желаемое состояние для приложения: образ, количество реплик, стратегия обновления. Под капотом управляет ReplicaSet.

ReplicaSet

Следит, чтобы было нужное число Pod. Обычно создаётся автоматически Deployment’ом.

Service

Стабильная “точка входа” для Pod по label-selector. Дает постоянный DNS/Virtual IP и балансировку между репликами.

  • ClusterIP: доступ внутри кластера
  • NodePort: порт на нодах (удобно локально)
  • LoadBalancer: внешний балансер (в облаке)

Ingress

Правила L7‑маршрутизации (HTTP/HTTPS) в кластер. Требует Ingress Controller (например, nginx-ingress).

Namespace

Логическая изоляция ресурсов. Удобно разделять окружения/проекты.

ConfigMap / Secret

Хранение конфигурации и секретов отдельно от образа. Можно подключать как env‑переменные или как файлы в volume.

Volumes

Диски/тома, которые монтируются в контейнер. В нашем манифесте используется emptyDir (временное хранилище на время жизни Pod).

Probes

Проверки здоровья: readinessProbe (готов ли принимать трафик) и livenessProbe (жив ли, нужно ли перезапустить). У нас — HTTP вызов /health.

kubectl: базовые команды

Почти всё в k8s — это YAML‑манифест + kubectl apply. Смотри состояние через get/describe и логи через logs.

Применить/удалить

kubectl apply -f k8s/
kubectl delete -f k8s/

Посмотреть состояние

kubectl get pods
kubectl get deploy
kubectl get svc
kubectl get events --sort-by=.metadata.creationTimestamp

Контекст (куда “смотрит” kubectl)

Если у тебя есть и docker-desktop, и kind, легко случайно деплоить “не туда”. Всегда проверяй текущий контекст. 

kubectl config current-context
kubectl config get-contexts
kubectl config use-context kind-kind

Диагностика

kubectl describe pod <pod-name>
kubectl logs -f <pod-name>
kubectl logs -f deploy/player-api
kubectl exec -it <pod-name> -- sh

Проброс порта (доступ с локальной машины)

kubectl port-forward svc/player-api 8000:80
curl http://127.0.0.1:8000/health

kind: локальный Kubernetes

kind (Kubernetes IN Docker) поднимает кластер k8s в Docker-контейнерах. Удобно для обучения/локальной разработки.

Создание и проверка

kind create cluster --name kind
kind get clusters
kubectl cluster-info

Локальные Docker-образы и kind

В kind ноды — это отдельный Docker‑контекст. Поэтому образ, собранный локально, нужно “загрузить” в кластер. 

export TAG="v1.1.1"
docker build -t player-api:$TAG .
kind load docker-image player-api:$TAG --name kind

Деплой нашего сервиса (Player API)

В репозитории уже есть манифесты: k8s/deployment.yaml и k8s/service.yaml.

Релиз через версию образа (рекомендуется)

export TAG="v1.1.1"
docker build -t player-api:$TAG .
kind load docker-image player-api:$TAG --name kind

kubectl config use-context kind-kind
kubectl apply -f k8s/deployment.yaml -f k8s/service.yaml
kubectl set image deployment/player-api player-api=player-api:$TAG
kubectl rollout status deployment/player-api

Authentication (JWT): как пользоваться

В уроке 1.1 добавлена базовая аутентификация: регистрация, логин и Bearer‑токен. Эндпоинты /players требуют заголовок Authorization.

1) Зарегистрироваться

curl -X POST http://127.0.0.1:8000/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"player1","email":"player1@example.com","password":"verysecret123"}'

2) Получить токен

Логин использует OAuth2 password flow и принимает application/x-www-form-urlencoded. 

curl -X POST http://127.0.0.1:8000/auth/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=player1&password=verysecret123"

3) Вызвать защищённый эндпоинт

# TOKEN=<access_token из /auth/token>
curl http://127.0.0.1:8000/players \
  -H "Authorization: Bearer $TOKEN"

Урок 1.3: Auth на уровне Ingress NGINX

Идея: отсеивать неавторизованные запросы до приложения, используя NGINX Ingress Controller. Это переносит “проверку входа” (есть ли токен / валиден ли токен) на edge‑уровень.

Authorization “по данным” (например, user может читать только себя) обычно остаётся в сервисе.

Схема: до (уроки 1.1–1.2) и после (1.3)

До Ingress весь HTTP‑трафик попадал в приложение (часто через kubectl port-forward на Service), и проверка JWT для /players выполнялась только внутри FastAPI. После 1.3 внешний HTTP сначала проходит через NGINX Ingress Controller: для защищённых путей выполняется subrequest на auth-url, и только при успешном ответе запрос уходит в тот же сервис.

Минимальные требования

  • GET /health доступен без авторизации
  • /players требует авторизацию на уровне Ingress

Подход A (минимальный): auth-url

NGINX Ingress умеет проксировать запрос на “auth endpoint” и решать, пропускать ли запрос дальше.

  • nginx.ingress.kubernetes.io/auth-url
  • nginx.ingress.kubernetes.io/auth-response-headers (опционально)

Подход B (расширенный): oauth2-proxy

Подходит, если хочешь OIDC/OAuth2 логин и централизованную авторизацию.

Проверка

# /health без токена
curl -i http://localhost/health

# /players без токена (ожидаемо 401/403 от Ingress)
curl -i http://localhost/players

# /players с токеном (должно дойти до сервиса)
curl -i http://localhost/players -H "Authorization: Bearer $TOKEN"

Урок 1.4: Admission Controllers (mutating / validating)

Идея: Kubernetes может перехватывать запросы к API до записи объекта в etcd и либо изменить объект (mutating), либо разрешить/запретить операцию (validating). На практике это чаще всего реализовано как MutatingWebhookConfiguration / ValidatingWebhookConfiguration.

В этом уроке мы используем Kyverno как “готовый” admission webhook‑стек: проще, чем с нуля поднимать TLS, Service и собственный webhook‑сервер.

Зачем это нужно (мотивация)

В уроках 1.1–1.3 мы учились защищать HTTP‑вход в сервис (JWT, роли, Ingress auth). Admission — про другую границу: Kubernetes API. Сюда приходят не только “внешние” запросы через Ingress, но и kubectl apply, Helm, CI и контроллеры GitOps — и все они в итоге создают/меняют объекты через apiserver.

Use case: заранее (до записи в etcd) нормализовать манифесты и/или запретить опасные/неприемлемые для платформы конфигурации. Это снижает класс ошибок “сначала выкатили, потом ловим в рантайме”, и помогает держать кластер предсказуемым для многих команд.

Типовые примеры: запрет :latest, обязательные labels для учёта/костов, дефолтные лимиты ресурсов, запрет привилегированных контейнеров, единообразные политики для CI и для людей (потому что оба пути проходят через один и тот же API).

Схемы: “без admission” vs “с admission webhooks”

Важно про порядок: на запрос сначала накладываются mutating изменения (патчи/инъекции), затем validating проверяет уже “почти финальный” объект. Поэтому валидация обычно отвечает на вопрос “можно ли сохранить то, что получилось после мутаций?”.

Задание

  • Установить Kyverno в kind.
  • Сделать mutating политику (добавить label).
  • Сделать validating политику (запретить :latest).

Установка Kyverno

kubectl create namespace kyverno || true
kubectl apply -f https://github.com/kyverno/kyverno/releases/download/v1.17.1/install.yaml

kubectl -n kyverno wait --for=condition=available deployment/kyverno-admission-controller --timeout=300s
kubectl -n kyverno get pods

kubectl get mutatingwebhookconfiguration,validatingwebhookconfiguration | grep -i kyverno || true

Mutate: добавить label

kubectl apply -f https://raw.githubusercontent.com/kyverno/policies/main/other/add-labels/add-labels.yaml
kubectl get clusterpolicy add-labels

Validate: запретить :latest (и включить Enforce)

kubectl apply -f https://raw.githubusercontent.com/kyverno/policies/main/best-practices/disallow-latest-tag/disallow-latest-tag.yaml
kubectl patch clusterpolicy disallow-latest-tag --type merge -p '{"spec":{"validationFailureAction":"Enforce"}}'
kubectl get clusterpolicy disallow-latest-tag

Проверка

# mutating: label появляется даже на server dry-run
kubectl create configmap kyverno-mutate-demo \
  --from-literal=k=v \
  --dry-run=server -o yaml | grep -F "foo: bar"

# validating: :latest должен быть отклонён API
kubectl create deployment kyverno-validate-demo --image=nginx:latest --dry-run=server

# validating: конкретный тег — ок
kubectl create deployment kyverno-validate-demo-ok --image=docker.io/library/nginx:1.27.0 --dry-run=server

Критерии готовности

  • Видны webhook‑конфигурации Kyverno.
  • Mutate реально меняет манифест (проверка через --dry-run=server).
  • Validate в режиме Enforce отклоняет :latest.

Если не запускается: чек-лист

Образ не подтягивается

  • Убедись, что сделал kind load docker-image ... (или используешь registry).
  • Посмотри: kubectl describe pod ... → события ImagePullBackOff/ErrImagePull.

Поды падают (CrashLoopBackOff)

  • kubectl logs -f deploy/player-api
  • kubectl describe pod ... (причина рестартов)

Не проходит readiness/liveness

  • Проверь, что /health доступен внутри контейнера и приложение слушает 0.0.0.0:8000.

Service не работает

  • Проверь selector/labels совпадают: kubectl get pods --show-labels
  • Проверь endpoints сервиса: kubectl get endpoints player-api

В Swagger не видно эндпоинты (например /auth/*)

  • Проверь, что ты смотришь на нужный инстанс: открой /openapi.json и поищи пути: curl -s http://localhost:8000/openapi.json | grep -Eo '\"/auth/[^\\\"]+'
  • Проверь, что выкатил новый образ в кластер (и правильный контекст): kubectl config current-context, kubectl get deploy player-api -o=jsonpath='{.spec.template.spec.containers[*].image}{"\\n"}'
  • Дождись завершения обновления: kubectl rollout status deployment/player-api
