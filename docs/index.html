Kubernetes (k8s): базовые сущности и инструменты

Короткая справка для запуска нашего FastAPI-сервиса в kind.

Сущности Kubernetes (что это такое)

Pod

Минимальная единица запуска. Обычно содержит 1 контейнер (но может быть несколько). У Pod есть IP внутри кластера, но он эфемерен — может пересоздаваться.

Deployment

Описывает желаемое состояние для приложения: образ, количество реплик, стратегия обновления. Под капотом управляет ReplicaSet.

ReplicaSet

Следит, чтобы было нужное число Pod. Обычно создаётся автоматически Deployment’ом.

Service

Стабильная “точка входа” для Pod по label-selector. Дает постоянный DNS/Virtual IP и балансировку между репликами.

  • ClusterIP: доступ внутри кластера
  • NodePort: порт на нодах (удобно локально)
  • LoadBalancer: внешний балансер (в облаке)

Ingress

Правила L7‑маршрутизации (HTTP/HTTPS) в кластер. Требует Ingress Controller (например, nginx-ingress).

Namespace

Логическая изоляция ресурсов. Удобно разделять окружения/проекты.

ConfigMap / Secret

Хранение конфигурации и секретов отдельно от образа. Можно подключать как env‑переменные или как файлы в volume.

Volumes

Диски/тома, которые монтируются в контейнер. В нашем манифесте используется emptyDir (временное хранилище на время жизни Pod).

Probes

Проверки здоровья: readinessProbe (готов ли принимать трафик) и livenessProbe (жив ли, нужно ли перезапустить). У нас — HTTP вызов /health.

kubectl: базовые команды

Почти всё в k8s — это YAML‑манифест + kubectl apply. Смотри состояние через get/describe и логи через logs.

Применить/удалить

kubectl apply -f k8s/
kubectl delete -f k8s/

Посмотреть состояние

kubectl get pods
kubectl get deploy
kubectl get svc
kubectl get events --sort-by=.metadata.creationTimestamp

Диагностика

kubectl describe pod <pod-name>
kubectl logs -f <pod-name>
kubectl logs -f deploy/player-api
kubectl exec -it <pod-name> -- sh

Проброс порта (доступ с локальной машины)

kubectl port-forward svc/player-api 8000:80
curl http://127.0.0.1:8000/health

kind: локальный Kubernetes

kind (Kubernetes IN Docker) поднимает кластер k8s в Docker-контейнерах. Удобно для обучения/локальной разработки.

Создание и проверка

kind create cluster --name kind
kind get clusters
kubectl cluster-info

Локальные Docker-образы и kind

В kind ноды — это отдельный Docker‑контекст. Поэтому образ, собранный локально, нужно “загрузить” в кластер. 

docker build -t player-api:latest .
kind load docker-image player-api:latest --name kind

Деплой нашего сервиса (Player API)

В репозитории уже есть манифесты: k8s/deployment.yaml и k8s/service.yaml.

Шаги

  1. Собрать образ player-api:latest.
  2. Загрузить образ в kind.
  3. Применить манифесты.
  4. Пробросить порт и проверить /health.
# 1) build
docker build -t player-api:latest .

# 2) load into kind
kind load docker-image player-api:latest --name kind

# 3) apply
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# 4) test
kubectl port-forward svc/player-api 8000:80
curl http://127.0.0.1:8000/health

Если не запускается: чек-лист

Образ не подтягивается

  • Убедись, что сделал kind load docker-image ... (или используешь registry).
  • Посмотри: kubectl describe pod ... → события ImagePullBackOff/ErrImagePull.

Поды падают (CrashLoopBackOff)

  • kubectl logs -f deploy/player-api
  • kubectl describe pod ... (причина рестартов)

Не проходит readiness/liveness

  • Проверь, что /health доступен внутри контейнера и приложение слушает 0.0.0.0:8000.

Service не работает

  • Проверь selector/labels совпадают: kubectl get pods --show-labels
  • Проверь endpoints сервиса: kubectl get endpoints player-api
